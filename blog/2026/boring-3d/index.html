boring-3D: Can Explicit 3D Supervision Help Latent World Models?

April 12, 2026

I wanted to see how to best incorporate vision signals into world models. Not in the obvious way — not "give the model a depth map as input and hope for the best." It was about implicits. How could we best induce an implicit understanding of geometry inside a model that only ever sees compressed latent vectors? The idea is that if you force a latent to predict depth alongside its normal job during training, the latent itself becomes geometrically richer. Then you throw the depth head away at inference. The geometry stays behind, baked into the representation.

So the question became: could I extract individual signals from robot videos — depth, point tracks, masks, centroids, shape descriptors — and finetune on them in isolation to find which ones actually help? That was the project.

Why LeWorldModel

I chose LeWorldModel (LeWM) for two reasons. First, JEPA is genuinely promising as a paradigm — Joint Embedding Predictive Architecture, predicting in latent space instead of pixel space. The model doesn't waste capacity reconstructing textures. It focuses on structure. Second, LeWM is small. ViT-Tiny encoder, about 7.6 million parameters, 192-dimensional latent. Small enough that I could actually iterate on it without needing a cluster.

The way it works: a ViT-Tiny encoder compresses each 224×224 RGB frame into a 192-dim CLS token. A 6-layer transformer predictor takes a context window of 3 latent frames plus actions and predicts the next latent. The model never generates images. It predicts embeddings. Loss is MSE in latent space.

That's both the strength and the limitation. You can't just look at the output and see if the model learned anything. To visualize, I built a retrieval pipeline — for each predicted latent, find the nearest real frame in the dataset by L2 distance. It's a proxy. The retrieved frame is a visual stand-in for whatever the model's internal state represents. More on that later.

Among the pretrained task weights available for LeWM, I went with the OG Cube weights because of 3D relevance — the cube task is synthetic manipulation with clear geometric structure, which seemed like the best starting point for a project about teaching geometry.

Why BridgeData V2

I needed real manipulation video. Not synthetic cubes that were too close to the source task, and not wild internet video that was too chaotic to extract clean signals from. BridgeData V2 sits in the right middle: real robot, real textures and clutter, tabletop kitchen manipulation, but structured enough that depth estimation and point tracking and segmentation all work end-to-end. It has 7-DOF action labels, state observations including gripper closure (useful for detecting contact events), and natural-language task descriptions I could filter on.

I grabbed about 100 episodes. 2,920 total frames. The episodes are short — 15 to 78 frames at roughly 5 FPS, so like 3 to 10 seconds each. That shortness is a real limitation, but it's also what made the extraction tractable.

Filtering for quality

I didn't just dump all 100 episodes into training. A huge part of making this work was choosing clips where extraction could actually succeed.

I built a weighted scoring system: task quality (prefer "put X on Y" and "pick up", penalize cloth folding), episode length (15–40 frames is the sweet spot), track displacement (prefer >60px total — if nothing moves much, what are we even learning?), mask quality (want 2–25% frame coverage with low temporal variance), depth quality (want >2.0 dynamic range). This gave me a pilot of 13 episodes, 334 frames. Carefully curated.

The full set kept all 100 episodes without filtering. That pilot-vs-full contrast ended up being one of the main findings of the whole project, but I didn't know that yet.

The extraction pipeline

The core engineering of this project was turning ordinary RGB robot video into structured packets of geometry-related signals. Each modality needed its own extraction step.

Depth: Video-Depth-Anything, ViTS encoder. Per-frame relative depth maps, temporally consistent. I used a scale-shift-invariant L1 loss for training, so relative depth is fine — no need for metric calibration.

Point tracking: CoTracker3, offline mode. 20×20 grid gives 400 tracked points per frame with visibility flags. Later I built a foreground-biased variant that places ~60% of points on the object region, which improved object coverage from roughly 5 to 47 visible object tracks per frame. That improvement mattered a lot for interpretation later.

Object masks: This was the hardest extraction by far — it gets its own section.

Centroids: Backproject masked depth to 3D using camera intrinsics, average the 3D points. One [x,y,z] per frame. Simple.

Shape descriptors: A 10D vector from the backprojected object point cloud — centroid xyz, PCA extents, spread ratios, depth range, visibility proxy. Designed to be compact enough for the model's bottleneck.

Pose estimation: I explored FoundationPose but it was blocked by CUDA toolkit and Visual Studio Build Tools dependencies. It just wouldn't install cleanly. I've since heard that pose estimation doesn't strictly need mesh priors anymore with some newer approaches, so this might not be as limited as I thought at the time. But it didn't make it into the finished pipeline.

The masking saga

Masking the manipulated object was supposed to be straightforward. It was not.

First I tried automatic motion-based prompting — pick the highest-displacement tracked-point cluster and feed it to SAM2. This almost always grabbed the gripper arm, because the arm moves the most. Obviously. Should have guessed that.

Then I tried a "stationary then moves" heuristic — find points that were still before contact and started moving after. Better, but SAM2 would sometimes latch onto the countertop near the object, or the destination container.

Then I tried text-guided segmentation — "the object being picked up" — using Moondream, SAM3, and Gemini. The science just isn't there yet for this specific use case. Too unreliable.

Then I tried using the contact signal from BridgeData as a segmentation prior. But the contact data is too coarse — it tells you when the robot is touching something, not which pixels are the object. Useful as a temporal prior ("look around this frame"), not as a mask.

The fix that actually worked: I built a tkinter annotation GUI where you click on the object in one frame, and SAM2 propagates the mask through the whole video. Manual, ugly, but it gave me trustworthy canonical masks for the pilot.

Packaging the data

All extracted signals were packed into HDF5 files: pixels, action, observation, depth, contact, tracks, track_visibility, object_mask, object_centroid, and later object_shape for the compact descriptor condition.

This sounds boring but it was one of the most useful things the project produced. It turned vague ideas about "using 3D in world models" into a concrete dataset format and training interface. How you format data to actually be used for these things is half the battle and nobody talks about it.

The auxiliary head setup

All models start from the same pretrained LeWM checkpoint. All finetuned on the same BridgeData. The only difference: which extra supervision head is active during training.

  • A: Baseline — no auxiliary head, just standard finetuning
  • B: Full-frame depth — predict a 56×56 depth map from the CLS token (~400K extra params)
  • C family: Masked depth variants — tight mask, dilated masks (8px, 20px), weighted full-frame with 5× boost on object pixels
  • D: Point tracks — predict 400×2 displacements from the CLS token (~135K extra params)
  • E: Depth + tracks together
  • F: Centroid — predict [x,y,z] from the CLS token (~37K extra params)
  • G: Shape descriptor — predict 10D object geometry summary (~37K extra params)

All auxiliary losses weighted at 0.1. Training: AdamW, lr=5e-5, batch 16, 20 epochs, bf16, 3–10 seeds per condition. I did change some encoding dimension architecture to support the auxiliary heads, but the core LeWM was kept intact.

This made the project a real controlled ablation. Same data, same base model, different extra supervision signal. Fair comparison.

Why full 3D was dropped

I need to be direct about this because it's important. The original entire purpose of this project was to test explicit 3D representations for world model training — point clouds, Gaussian splats, maybe NeRF-like structure. That was the dream.

It got deferred. Not because the idea is wrong, but because this specific setup is the wrong place to test it.

Static monocular cameras do not allow 3D vision to shine. BridgeData has a fixed overhead camera. Gaussian splatting needs multi-view or camera motion. NeRF needs the same. Two frames from the same fixed camera give you the same point cloud twice — there's no real geometric information gain frame to frame. Monocular depth gives you shape up to an unknown scale factor, which is useful for auxiliary supervision but not for real 3D reconstruction.

So the project became "boring 3D signals" — depth, tracks, masks, centroids, compact shape descriptors. The things I could actually extract cleanly and test fairly. The full explicit 3D work is the focus of a future project, but it needs different data.

The harness can handle very limited point cloud work though, and I should show that — the backprojected object clouds look reasonable frame-by-frame, they're just not multi-view reconstructions.

Pilot results

On the curated 13-episode pilot (334 frames), things looked genuinely promising:

ConditionSignalvs Baseline
BDepth-11.5%
C4Weighted depth-14.5%
EDepth+Tracks-14.0%
GShape (10D)-6.1%
DTracks alone-4.9%
FCentroid-3.0%

Depth worked. Depth-based combinations were the strongest. Models genuinely beat the static baseline on episodes with visible object motion. This wasn't noise — the models learned dynamics.

The mask family sweep

I tried different sizes of masks to see whether focusing depth supervision on the object or allowing visualization of the immediate surroundings would help.

ConditionApproachvs Baseline
BFull-frame depth-10.5%
C4Full-frame + 5× weight on object-10.9%
C1Tight object mask only-7.5%
C28px dilated mask-5.8%
C320px dilated mask-2.3%

The pattern is monotonic and kind of surprising: the more you restrict depth supervision to the object region, the worse it gets. The reason is the scale-shift-invariant normalization — when you normalize within a tiny masked region (~2% of pixels), the scale and shift estimates become noisy. Full-frame normalization is more stable. If you're using scale-shift-invariant depth loss, just don't mask it.

The tracks result

Tracks were one of the most interesting negative results.

First round with uniform tracking — ~5 object tracks per frame — they did nothing. Okay, maybe the data quality was bad.

So I built foreground-biased tracking: ~47 object tracks per frame. The visual audit confirmed the tracks genuinely landed on the object. They looked good.

They still did nothing.

This matters because it means the failure is architectural, not about data quality. A single 192-dim CLS token cannot usefully encode 400 independent 2D displacements through a flat MLP. The architecture is the bottleneck. There's recent work on trajectory-token approaches that treat each track as its own token with local visual features and occlusion handling — that's a fundamentally different architecture, and it seems to work. Tracks aren't dead as a concept. They're dead in this formulation.

The sign flip at full scale

Then I ran the full 100-episode set. And the story flipped.

ConditionPilot (13 ep)Full (100 ep)
B Depth-11.5%+1.0%
E Depth+Tracks-14.0%+1.3%
D Tracks-4.9%+0.2%

The advantages didn't just shrink. They reversed direction. At 100 episodes, the baseline learned everything the auxiliary heads were trying to teach, and the extra loss terms became regularization noise. Absolute MSE dropped about 4× from pilot to full, confirming the model had much more to learn from the larger set. The auxiliary heads acted as inductive bias — useful when data was scarce, a burden when there was enough diversity to learn the structure end-to-end.

This is the main finding of the project. Not "3D helps" or "3D doesn't help." It's: explicit 3D supervision can act as a useful low-data inductive bias, but in this CLS-only setup it does not survive broader finetuning. The sign flip is clean: -14% at 13 episodes, +1% at 100. Same architecture, same loss, same optimizer. Only variable is data scale.

What is CLS and why does it matter so much here?

This needs explaining because it came up constantly.

LeWM's ViT encoder produces a CLS token AND a full 16×16 grid of 196 patch tokens. The patch tokens carry spatial layout information — where things are, what's next to what. But the prediction path only uses the CLS token. The 196 patch tokens are discarded before prediction.

So the entire scene — every object, every spatial relationship — has to compress into one 192-dimensional vector. That's the bottleneck.

This explained almost everything:

  • Why full-frame depth could still help (depth is a global scene property — near vs far — that maps well to a single embedding)
  • Why raw point tracks couldn't help (400 independent local displacements can't squeeze through one global vector and a flat MLP)
  • Why tiny compressed signals like centroid were too weak (not enough gradient signal from 3 numbers to reshape a 192-dim space)
  • Why "explicit 3D" isn't just about extraction quality — the architecture has to be able to use the signal

Probably the single biggest lesson of the whole project.

Evaluation and visualization

Eval suites are huge. When I was stuck — and I was stuck a lot — the thing that got me unstuck was always: evaluate and visualize what your data and results actually look like. Stop guessing. Look.

The packet viewer was one of the smartest things the project grew into. For each episode I could inspect RGB, depth, track overlays, mask overlays, object-only depth, scene point cloud, object point cloud. This stopped the project from being pure metric worship. When the 10-episode audit showed depth looked plausible, tracks landed on objects, masks were on the right things, and object clouds were small but recognizable — that meant later negative results couldn't be blamed on broken extraction.

Retrieval rollout dashboards showed side-by-side GIFs of ground truth vs static baseline vs model predictions via nearest-neighbor retrieval. I had two retrieval modes: global search across all frames (honest but choppy) and same-episode search (smooth and restricted). Distance-over-time plots for each model. Auto-curation tags that labeled episodes as "static-dominates" or "models-beat-static" or "tight-tie" — savvy quality-of-life stuff for finding the best presentation examples quickly.

The decoder. I did train a cross-attention decoder to map latents back to RGB. It was blurry. Patchy grids, objects dissolving into mush. 50 epochs on 334 frames is not enough. I'm showing it anyway because it's honest about where the project's visualization limits are, and because the retrieval approach is the real evidence.

What I built — the infrastructure

Beyond the experiment results, the project produced a bunch of reusable stuff:

  • A LeWM finetuning harness. Config-driven, supports arbitrary auxiliary heads. I released a standalone decluttered version of this so other people can finetune LeWM on their own data without inheriting all my experimental clutter.
  • A structured 3D extraction pipeline. Depth, tracks, masks, object clouds, centroids, shape descriptors, from monocular robot video.
  • Qualitative set filtering. The weighted scoring system for episode selection.
  • HDF5 dataset packaging for multi-signal robot video.
  • A packet visualization and auditing system.
  • Retrieval-based rollout visualization for latent world models that don't generate pixels.

These are useful regardless of whether depth won or lost.

Things to note

CLS tokens absorb dense global signals like depth way better than spatially local signals like point tracks. This sounds obvious in retrospect but I had to learn it the hard way. Depth is "the whole scene is near or far" — that compresses into one vector fine. Point tracks are "this specific point moved 3px right and that specific point moved 7px up" — that doesn't compress well at all.

Even if a signal is extracted well, the architecture needs to be friendly to it. I improved track quality a lot. It didn't matter. The bottleneck was the model, not the data.

Curation was the biggest value add. The carefully filtered pilot was where everything looked promising. The unfiltered full set is where the gains disappeared. That's partly about data quality and partly about the marginal value of auxiliary supervision decreasing with more data. Either way — curation mattered more than I expected.

Limitations

I want to be clear about these:

  • Short videos. 3–10 seconds each. Long-horizon dynamics barely exist in this data.
  • Small set. Even 100 episodes is modest. This is small-data → modest-data, not small → large.
  • Limited visual detail. 224×224, overhead view, similar-looking kitchens.
  • Small model. ViT-Tiny, 192-dim latent, 7.6M parameters. A bigger model might have enough capacity to benefit from auxiliary signals even with more data.
  • Not measuring downstream task success. I'm measuring latent prediction quality (pred_mse), not policy performance or manipulation success rate. The real test would be whether better latents → better robot behavior.
  • No pose estimation, no explicit 3D. The full splat/NeRF/pose pipeline was deferred.
  • Decoder was too weak to be primary visual evidence.
  • Full-scale only ran A, B, D, E. Not all conditions got the full 100-episode treatment.
  • The CLS bottleneck is very specific to this architecture. A patch-aware predictor or a model that doesn't throw away spatial tokens might tell a completely different story.

What's next

I want multi-view or moving camera data. Wrist cameras, maybe. This would unlock real Gaussian splatting and NeRF experiments. The original project vision requires it. But I'm also thinking about how to enable training on wide swaths of web video, which is going to be monocular and mobile almost certainly. So the pipeline needs to work for both.

Trajectory tokens. Instead of predicting 400 displacements from one CLS vector, treat each track as its own token with local visual context. The literature suggests this works. It's a fundamentally different architecture and probably the right way to make tracks useful.

Patch-aware predictor. If the predictor could read the 196 patch tokens instead of just CLS, spatially rich signals might actually get through the bottleneck.

Scale vs curation. The interaction between inductive bias and data scale is the real open question. Does the sign flip hold at 1000 episodes? At 10,000? With a bigger model? The finding is real in this regime but I don't know how general it is.

The takeaway

I built a LeWorldModel finetuning and 3D extraction testbed on real robot manipulation video. I tested which explicit 3D signals help latent prediction. On a tiny curated pilot, depth-based supervision gave real gains. On a broader 100-episode run, those gains disappeared or reversed. The auxiliary heads became noise generators the more data there actually was.

The result is a clean scaling paradox: explicit 3D supervision works as a low-data inductive bias, but in this CLS-only setup it doesn't survive broader finetuning. The project also showed that signal quality alone isn't enough — the architecture has to be able to use the signal, and the form of supervision matters as much as the semantic content.

This isn't a success story about 3D helping world models. It's a documented experiment with real infrastructure, real extraction, real ablations, and honest results. The sign flip is the finding. Same architecture, same loss, same optimizer, same data source. Only variable is data scale. That's reproducible and worth knowing about.
